## 2. Architectural Anchors

These three decisions are load-bearing for the implementation plan; later sections build on them.

### 2.1 Dispatch lifecycle for `/<pack>:<command> args`

1. **Parse.** The slash dispatcher recognises `^/([a-z][a-z0-9-]*):([a-z][a-z0-9-]*)(?:\s+(.*))?$` as a namespaced pack command. Anything else falls through to orchestrator built-ins (currently `/spore list|pin|unpin`).
2. **Resolve.** Look up `<pack>` in `SporeRegistry`. Three failure modes:
   - Pack not found at any tier ŌåÆ "no spore named `<pack>`".
   - Pack found, command file missing ŌåÆ "spore `<pack>` has no command `<name>`".
   - Pack found, command found, but pack is not the active spore ŌåÆ refuse with "pin it first via `/spore pin <pack>`".
3. **No auto-germination.** A non-active pack's command never silently switches the active spore. This avoids the hidden-state trap that `/spore unpin` exists to fix.
4. **Body expansion.** Strict Claude Code convention. If the body contains `$ARGUMENTS`, replace it exactly with the raw arg text (which may be the empty string when the user invokes the command bare). If the placeholder is absent, append `"\n\n" + args` to the body unconditionally ŌĆö even when args is empty, yielding a trailing `"\n\n"`. Authors who want to avoid the trailing newlines on bare invocation should use `$ARGUMENTS` explicitly.
5. **Inject as user message.** The expanded body becomes the next user message. The model sees it as if the user typed it. The ReAct loop runs naturally; `spawn_subagent` calls inside the body work because the pack is active. The expanded body is **never re-dispatched** even if it starts with `/` ŌĆö the dispatcher fires only on input read directly from the REPL, not on synthesised user messages. This keeps the dispatch tree flat (no recursive slash chains) and removes a class of confusing author footguns.
6. **Conversation log.** Record the **expanded body** as the user message in `.myceliate/history/<session>.jsonl`, not the raw slash. `--resume` rehydrates exactly what the model previously saw, so re-running on a session whose command markdown was edited between sessions remains idempotent.
7. **Audit log.** The raw `/<pack>:<command> args` is logged at info level for traceability ŌĆö separate from the conversation log.

### 2.2 System-prompt composition

The hardcoded `descriptionsSection` injection at `index.ts:271ŌĆō275` extracts into a section composer that advertises:

- All loaded sectors (current behaviour, unchanged).
- **Only the active spore's commands** ŌĆö listing every loaded pack's commands would bloat the prompt and tempt the model to suggest unreachable commands.

Active-spore swaps re-run `replaceGerminatedSection` (already shipped in Phase 21 stretch); the new germinated body advertises the new spore's command set. The state machine stays clean.

### 2.3 Tool restriction enforcement

- **Manifest field.** `myceliate.yaml` gains `allowed_tools: string[]`, additive and optional. Absent ŌåÆ current behaviour (model sees all registered execution tools).
- **Scope.** Filters the orchestrator's tool list while the pack is the active spore. Coordination tools (`germinate_spore`, `spawn_subagent`) are never restrictable; they live outside the manifest's allowlist field per R9 partition.
- **Enforcement point.** **Registration-time filtering**, not dispatch-time gating. When the active spore changes, the registry exposes a filtered view; denied tools are removed from the JSON Schema sent to the API. The model never sees a denied tool, never tries to call one, and we avoid the ReAct-loop pathology of repeated denied calls.
- **Sub-agent inheritance ŌĆö deferred.** Sub-agents continue to receive the persona's full execution tool set per current R9. Per-persona allowlists (the "sector-aware sub-agent capability scoping" item from v1.3 spec ┬¦13) stay deferred to v1.5.
- **Resume safety.** Historical `tool_call` messages in `--resume`-rehydrated history that reference tools now denied by the active manifest are safe. The DeepSeek/V3 message format permits historical tool executions in the `messages` array without requiring the same tool to appear in the request's `tools` array. The model won't attempt to call those tools on subsequent turns because they're absent from the schema sent for the current turn. This is documented as known, safe behaviour.

## 3. Per-Pack Slash Commands (Phase 22 scope)

### 3.1 Disk layout

```
<pack>/
  myceliate.yaml         # manifest (existing)
  SKILL.md               # sector body (existing)
  agents/<persona>/SKILL.md   # personas (existing)
  commands/<name>.md     # NEW ŌĆö flat .md files; one command per file
```

`commands/` lives alongside `agents/` under the pack root. Discovered at boot by an extension to `bootSpores` that walks the directory and parses each `*.md` via the existing `parseSkillFrontmatter` helper (the DRY-extracted utility from Phase 19).

Subdirectories under `commands/` are ignored. Non-`.md` files are ignored. Malformed frontmatter (missing `name` or `description`) emits a boot warning and the file is skipped ŌĆö same posture as the persona loader.

### 3.2 Frontmatter

```markdown
---
name: lit-review
description: One-shot literature review template. Synthesises 5 sources into a structured brief.
argument-hint: <topic>
---

# Literature Review

[body ŌĆö the prompt that fires when /<pack>:lit-review is dispatched]
```

Required keys: `name`, `description`. Optional: `argument-hint` (for future `/help` UX). Filename basename must equal frontmatter `name`; mismatch emits a boot warning and the file is skipped.

### 3.3 Body semantics

The body is a templated **user-message expansion**, not a system prompt. Authors must write commands as user-prompts ("Synthesise the topic below into a literature review with 5 sources..."), not as system-prompts ("You are a literature reviewer..."). The spec explicitly documents this divergence from the persona authoring pattern to prevent ReAct-loop hallucinations: a body written in second-person system-prompt voice, when injected as a user message, confuses the model about who is speaking and corrupts the loop.

`$ARGUMENTS` placeholder is substituted with the parsed args text ŌĆö exactly. If the body contains no `$ARGUMENTS`, the args are unconditionally appended after `"\n\n"` (even when args is empty, in which case the append produces a trailing `"\n\n"`). Use `$ARGUMENTS` explicitly when bare-invocation cleanliness matters; use the implicit-append form when args naturally trail the template.

### 3.4 Discovery

`SporeRegistry` is extended to track per-pack command sets. After Phase 22:

```ts
type SporeRecord = {
  manifest: SporeManifest;
  bodyPath: string;
  agents: PersonaRecord[];
  commands: CommandRecord[];   // NEW
};

type CommandRecord = {
  name: string;
  description: string;
  argumentHint?: string;
  bodyPath: string;
};
```

Discovery happens once at boot via `bootSpores`. Hot-reload is out of scope (carries forward the v1.3 deferral).

### 3.5 Slash dispatcher

The current inline `/spore` dispatch at `replSession.ts:67ŌĆō95` is extracted into `src/cli/slashDispatcher.ts`. The dispatcher exposes a single `dispatch(input, context)` entry point that:

1. Tries the namespaced regex. Match ŌåÆ resolve via `SporeRegistry`, run ┬¦2.1 lifecycle.
2. Falls through to orchestrator built-ins (`/spore *`) handled by the existing `sporeSlashCommands` module.
3. Returns a discriminated union:

```ts
type DispatchResult =
  | { kind: 'expanded-prompt'; body: string }       // pack command ŌĆö re-inject as user message
  | { kind: 'orchestrator-output'; text: string }   // built-in like /spore list ŌĆö print via onSlashOutput
  | { kind: 'no-match' };                           // not a slash command ŌĆö pass through as user prompt
```

The REPL session converts these into either a re-dispatched user prompt (kind `expanded-prompt`), a printed orchestrator output (kind `orchestrator-output`), or an unmatched-input passthrough (kind `no-match`).

### 3.6 System-prompt composition

The hardcoded `descriptionsSection` at `index.ts:271ŌĆō275` extracts into `src/orchestrator/composeSystemSections.ts`:

- **Sectors** ŌĆö list every loaded sector's `name`, `accent_color`, `description`. Same as today.
- **Active commands** ŌĆö if `activeSpore` is set, list its commands as `<pack>:<name> ŌĆö <description>` so the model can suggest them in conversation. Empty when no spore is pinned (`--no-spore` or pre-germination).

The composer's output is the body of the germinated system-prompt section, replaced via `replaceGerminatedSection` on each spore swap. Same composer is also called once at boot for the initial render.

## 4. Per-Pack Tool Restrictions (Phase 23 scope)

### 4.1 Manifest schema extension

`myceliate.yaml` gains an optional `allowed_tools: string[]` field:

```yaml
name: research
description: Sector pack for academic research workflows.
version: 1.0.0
accent_color: "#4a90c4"
allowed_tools:
  - read_file
  - grep
  - list_dir
keywords: [research, paper, citation]
agents: [literature-reviewer, citation-checker, methods-critic]
```

Semantics:

- **Absent / undefined** ŌåÆ current behaviour. Model sees every execution tool registered to the orchestrator.
- **Empty array `[]`** ŌåÆ strictly zero execution tools. The orchestrator retains only its coordination tools (`germinate_spore`, `spawn_subagent`). Valid by design ŌĆö supports tightly-scoped reflection or planning packs.
- **Names not present in the registry** ŌåÆ fail-soft. The Logger DI emits a boot warning, the unknown name is dropped from the active set, and boot continues. Never crashes the orchestrator.
- **Coordination tool names** (`germinate_spore`, `spawn_subagent`) ŌåÆ silently stripped with a warning. They aren't restrictable; including them is an authoring error.

**Allowlist (not blocklist).** v1.4 deliberately ships an allowlist model rather than a `denied_tools` blocklist. Blocklists are a fail-open capability-security anti-pattern: any new tool added to the global registry would default to allowed for every existing pack manifest, silently widening capabilities behind authors' backs. The allowlist's verbosity (re-stating tools per pack) is the price of fail-closed correctness.

The Zod schema in `src/spores/SporeManifest.ts` is the source of truth (R3); JSON Schema is derived.

### 4.2 Tool registration extraction

The current tool-registration block at `index.ts:201ŌĆō255` extracts into `src/runtime/bootTools.ts`. The function returns the populated `ToolRegistry` and a `setActiveSpore(name | null)` callback that recomputes the visible tool set when the active spore changes.

The visible tool set is the *intersection* of:

- The registered execution tools.
- The active spore's `allowed_tools` field, if defined.
- Always: the coordination tools.

If no spore is active (`--no-spore` or pre-germination), all registered execution tools are visible (current behaviour).

### 4.3 ToolRegistry filtered view

`ToolRegistry` gains two new methods:

- `getActiveTools(): Tool[]` ŌĆö returns the filtered set. The adapter (V3 / V4) calls this when assembling the API request's `tools` array.
- `setActiveAllowlist(names: string[] | null): void` ŌĆö updates the active allowlist. `null` resets to "no filter".

Internal lookups by tool name (when a `tool_call` result comes back during a turn) bypass the filter and use the full registry. Historical tool executions remain valid even if the current allowlist excludes them ŌĆö see ┬¦2.3 resume safety.

### 4.4 Active-spore change notification

When `germinate_spore`, `/spore pin`, or `/spore unpin` changes the active spore (the unpin case sets it to `null`):

1. `setActiveSpore(name)` is called on the `bootTools` return object.
2. `setActiveAllowlist` propagates to `ToolRegistry`.
3. The next API request's `tools` array reflects the new allowlist.
4. The system-prompt composer (┬¦3.6) re-renders the germinated section.
5. The UI's `<GerminationCard>` shows the new spore's accent colour.

All five steps are coordinated through the existing germination event path; no new events are introduced.

## 9. Migration & Backward Compatibility

v1.4 is **fully additive**. All v1.3 spore manifests, persona files, conversation logs, and `.myceliate/` state continue to work without change:

- `commands/` directory is optional; spores without one have an empty `commands` field and behave identically.
- `allowed_tools` field is optional; manifests without it inherit the existing "all execution tools visible" behaviour.
- v1.3 conversation logs replay correctly under v1.4 because body-expansion logic only applies to *new* slash dispatches; historical user messages pass through as-is.
- The new system-prompt composer produces identical output to v1.3's hardcoded section for any v1.3-shaped pack (no commands ŌåÆ no command listing).

## 13. Open Questions

All architectural decisions for v1.4 are locked. The implementation plan (`docs/superpowers/plans/v1.4-maximal-sprint.md`) breaks these into TDD phases, branches, and per-task tests; it does not introduce new architectural choices.

Resolutions during spec review (2026-05-09):

- **`/spore tools` phase home** ŌåÆ Phase 23 task 5. Built alongside the allowlist logic it introspects.
- **Allowlist vs blocklist** ŌåÆ allowlist only. Blocklists are a fail-open capability-security anti-pattern: new tools added to the registry would default to allowed for every existing manifest. See ┬¦4.1 for the locked language.
- **Empty `allowed_tools: []`** ŌåÆ strictly zero execution tools; coordination tools retained. Valid by design.
- **Boot warning surface for unknown allowlist names** ŌåÆ fail-soft. Logger DI emits a warning, the unknown name is dropped from the active set, boot continues. Never crashes startup.
- **`$ARGUMENTS` whitespace** ŌåÆ strict Claude Code convention. Placeholder present: exact replacement (raw args, may be empty). Placeholder absent: unconditional `"\n\n" + args` append (yielding trailing `"\n\n"` when args is empty). Authors who want bare-invocation cleanliness use `$ARGUMENTS` explicitly.
