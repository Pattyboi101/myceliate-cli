# Myceliate v1.5 — Mycelium Sprint

## 1. Overview & Goal

v1.5 matures the Myceliate network through four sequential phases. The sprint takes its name from the mycelial network itself — an underground web of hyphae that connects, routes, digests, and remembers. Each phase wires another layer of that network into the existing v1.4 architecture, all strictly additive — R1–R12 stay intact.

The four phases:

- **Phase 1 — Hyphal Continuity.** Fuse the orchestrator and BullMQ worker into a single managed hyphal link. The orchestrator owns the worker process lifecycle, eliminating the v1.4 walk-point 3 failure where a forgotten `pnpm queue:worker` silently broke bash execution. Preserves R6 (BullMQ for >500ms operations) intact; only changes who starts the worker and how its lifecycle is bounded.

- **Phase 2 — Anamorph Routing.** Add a Spore-role-to-model dispatcher. Mechanical/cloned execution paths (Anamorph, V4-Flash) and generative/architectural paths (Teleomorph, V4-Pro) get routed deterministically. Cuts bulk operational cost ~92% for terminal-heavy workloads while preserving reasoning quality on planning turns.

- **Phase 3 — Exoenzyme.** Translate external MCP server schemas into lightweight Markdown skills before the agent ingests them — sidecar processes that digest raw schema noise so the Spore stays thin. Target servers: Playwright (browser automation) and Postgres (DB introspection) as proof-of-concept.

- **Phase 4 — Anastomosis.** Agent-controlled `compact_state` tool for semantic handoffs. The Spore writes a structured summary to `.myceliate/memory/anchors.md` at logical task boundaries; the platform safely clears the context window and re-injects the summary as a fresh anchor. Replaces R10's automatic platform-side compaction with agent-aware semantic compaction.

This spec covers Phase 1 in implementation detail. **Phases 2–4 are scoped at the architectural-shape level here and will receive their own design amendments before each phase begins.** The intent is to ship Phase 1 fast, then design and implement the routing/MCP/anastomosis trio sequentially through v1.5.

The strategic research brief (`docs/Autonomous-Agent-CLI-Research-Brief.docx`) informed this sprint shape. The brief recommended Phase 1 = Tiered Routing, but codebase verification showed the routing infrastructure already exists in v1.4 (`model` is plumbed end-to-end through adapters, `subagentLoop`, `replSession`, `index.ts`, `QueryEngine`); only a Spore-role dispatcher is missing. That dispatcher becomes Phase 2. Phase 1 instead addresses a concrete v1.4 lived-experience gap: walk-point 3 silently failing because the BullMQ worker wasn't running in another terminal.

**Out of scope and deferred to v1.6+ or later** (full list in §8): cron spores / OpenClaw daemon (research brief Vector 8); FTS5 episodic memory layer (Vector 7); P2P agent collaboration (Vector 5); parallel message buses (Vector 4); MicroVM sandboxes (Vector 9); raw-vision multi-modal context (Vector 10).

## 2. Architectural Anchors

Three decisions are load-bearing for the Phase 1 implementation; later phases extend these but don't break them.

### 2.1 Orchestrator owns the BullMQ worker lifecycle

Currently the worker runs in a separate user-managed terminal via `pnpm queue:worker`. The orchestrator submits jobs via `getQueue().add()` and consumes results via BullMQ's `completed`/`failed` events. This worked through v1.4 — until walk-point 3 smoke revealed a UX gap: a forgotten worker process silently broke bash execution.

In v1.5 the orchestrator becomes the lifecycle owner. At boot it spawns the worker as a managed child process via `child_process.fork`, routes its stdio to a log file, and tears it down on orchestrator exit. **The worker's job-processing logic in `src/queue/worker.ts` is unchanged — only its startup harness, signal handling, and shutdown hooks change.** The Worker class instantiation, the BullMQ event wiring, and the bash command execution all stay exactly as they are in v1.4.

This preserves R6 (BullMQ for heavy I/O) intact. The architectural seam between orchestrator and worker still exists; only its operational presentation changes from "two terminals" to "one terminal".

### 2.2 Pending-jobs registry for crash recovery

The orchestrator's bash tool returns a Promise resolved by BullMQ's `completed` event. If the worker crashes mid-job, BullMQ's default behaviour is to wait for the job lock to expire (~30s) before marking the job stalled. From the user's perspective, the CLI hangs.

To eliminate this hang, the workerLifecycle module maintains a `Map<jobId, { reject: (err: Error) => void }>` of in-flight bash promise rejecters. When `child.on('exit')` fires with a non-zero code (or any exit while jobs are pending), the lifecycle iterates the Map and rejects all pending promises with `WorkerCrashedError`. The orchestrator's ReAct loop surfaces the crash to the user immediately — typically within 100ms — rather than waiting on BullMQ's stall watcher.

`Map`, not `WeakMap`, because the key is a primitive `jobId: string`. Entries are explicitly released on completion or failure via `releaseJob(jobId)`.

### 2.3 Fail-fast on missing infrastructure

A pre-flight `redis.ping()` runs before the worker is spawned. If Redis is down, the CLI exits with a clean diagnostic — `myceliate: Redis is required for bash execution. Run docker compose up -d redis and try again.` — rather than spawning a worker that crashes immediately on connection failure.

We explicitly do **not** auto-restart the worker on crash. At single-developer scale, auto-restart silently masks environment rot (Redis container drift, OOM patterns, version mismatches). Fail-fast surfaces these conditions where the user can address them. The deferred future is an explicit `myceliate doctor` subcommand that diagnoses worker-process state on demand; auto-restart can be reconsidered if real usage shows the failure modes are predominantly transient.

## 3. Phase 1 — Hyphal Continuity

### 3.1 New module: `src/queue/workerLifecycle.ts`

A single new file owns the spawn/health/cleanup state machine. Co-located with `worker.ts` because the lifecycle is intrinsically tied to the BullMQ worker and Redis connection — extracting a separate `src/process/` or `src/boot/` directory for one lifecycle manager creates premature fragmentation. Estimated size: 150–200 LOC for the module; unit + integration tests are separate (see §3.6).

**Public API (TypeScript surface):**

```ts
export interface WorkerLifecycle {
  /** Pre-flight Redis ping + spawn the worker child process. */
  start(): Promise<void>;

  /** Register a pending job; lifecycle calls reject() on worker crash. */
  trackJob(jobId: string, reject: (err: Error) => void): void;

  /** Job completed/failed normally — drop from pending registry. */
  releaseJob(jobId: string): void;

  /** Send SIGTERM, then SIGKILL after timeout. Idempotent. */
  shutdown(): Promise<void>;
}

export class RedisUnavailableError extends Error {
  constructor(cause?: Error) {
    super(
      'myceliate: Redis is required for bash execution. ' +
      'Run `docker compose up -d redis` and try again.',
      { cause },
    );
    this.name = 'RedisUnavailableError';
  }
}

export class WorkerCrashedError extends Error {
  constructor(
    public readonly exitCode: number | null,
    public readonly signal: NodeJS.Signals | null,
  ) {
    super(`BullMQ worker crashed (code=${exitCode}, signal=${signal})`);
    this.name = 'WorkerCrashedError';
  }
}

export function createWorkerLifecycle(opts: {
  redis: Redis;
  logger: Logger;
  workerScript?: string;   // injectable for tests; defaults to import.meta.url-relative './worker.js' (see §3.2 step 2)
  killTimeoutMs?: number;  // injectable for tests; defaults to 2000
}): WorkerLifecycle;
```

### 3.2 Boot sequence (in `src/index.ts`, before REPL init)

Inserted after the existing logger/Redis setup, before the orchestrator and Ink renderer mount:

```ts
const workerLifecycle = createWorkerLifecycle({ redis, logger });
await workerLifecycle.start();   // throws RedisUnavailableError → main() catches and exits 1
```

Steps inside `start()`:

1. **Redis pre-flight.** `await redis.ping()` with a 2s timeout. On failure: throw `RedisUnavailableError`. The caller in `main()` catches it, prints the diagnostic, and exits 1.

2. **Fork the worker.** `child_process.fork(workerScript, [], { stdio: ['ignore', 'pipe', 'pipe'], env: process.env })`. Stdin ignored (the worker takes no user input); stdout and stderr piped so we can route them to disk. **Worker script resolution:** the lifecycle module resolves the worker via `fileURLToPath(new URL('./worker.js', import.meta.url))` — sibling-path relative to itself. In built/prod mode this resolves to `dist/queue/worker.js` and forks under bare `node`. In dev mode (orchestrator launched via `pnpm dev` / `tsx src/index.ts`), the resolved path is the `.ts` source; the fork must inherit a TypeScript loader. Implementation choice: detect dev mode (e.g. via `process.execArgv.some(a => a.includes('tsx'))` or `MYCELIATE_DEV=1` env var set by the `dev` script) and pass `execArgv: ['--import', 'tsx']` to the fork in that case. Defer to the implementation plan for the exact detection heuristic.

3. **Route stdio.** Open `.myceliate/logs/worker.log` as an append stream (created if missing). Pipe `child.stdout` and `child.stderr` to it. The Ink UI never sees worker output. Honours U4 (logger never writes to stdout while Ink is mounted) and prevents the TUI corruption failure mode flagged during research review.

4. **Register process exit hooks.** `process.on('exit')`, `process.on('SIGINT')`, and `process.on('SIGTERM')` all call `workerLifecycle.shutdown()`. Inside `shutdown()`: `child.kill('SIGTERM')`; if the child hasn't exited within `killTimeoutMs` (default 2s), `child.kill('SIGKILL')`. The method is idempotent — subsequent calls after the first successful kill are no-ops, so stacked signals don't double-kill or throw.

5. **Register crash handler.** `child.on('exit', (code, signal) => ...)`. The lifecycle module maintains a private `shutdownInitiated: boolean` flag, set to `true` at the top of `shutdown()`. The exit handler distinguishes:
   - `shutdownInitiated === true` → clean shutdown path. Log `{ event: 'worker_shutdown_complete', code, signal }`. No promise rejection.
   - `shutdownInitiated === false` AND (`code !== 0` OR pending jobs exist) → crash path. Iterate the pending-jobs Map and call each `reject(new WorkerCrashedError(code, signal))`. Log a structured event (`{ event: 'worker_crashed', exitCode: code, signal, pendingJobs: [...keys] }`) so the next ReAct iteration surfaces the crash cleanly via the orchestrator's existing error-handling path.

### 3.3 Pending-jobs registry integration

The existing bash tool dispatcher (in the queue→tool bridge) already returns a Promise that resolves when BullMQ emits `completed` for the corresponding `jobId`. Modify the dispatch wrapper at the call site to:

1. Call `workerLifecycle.trackJob(jobId, reject)` immediately after `queue.add()`.
2. In the existing `.on('completed')` and `.on('failed')` handlers, call `workerLifecycle.releaseJob(jobId)` before resolving/rejecting the user-facing promise.

The bridge module that currently subscribes to BullMQ events will need a small refactor to thread the `workerLifecycle` reference through. This is the only orchestrator-side code touched by Phase 1 outside `src/index.ts` and `src/queue/`.

### 3.4 Worker-side: graceful shutdown signal handling

The `src/queue/worker.ts` entry point must respond to `SIGTERM` by closing the BullMQ Worker cleanly: drain in-flight jobs back to the queue (BullMQ's `Worker.close()` does this), close Redis connections, exit 0. Without this, the orchestrator's `SIGTERM` → `SIGKILL` escalation triggers on every clean shutdown, producing noisy log output and potentially leaking jobs that were mid-execution.

```ts
// in src/queue/worker.ts main()
process.on('SIGTERM', async () => {
  logger.info({ event: 'worker_sigterm_received' });
  await worker.close();   // BullMQ Worker.close() drains active jobs and disconnects
  process.exit(0);
});
```

This is the only logic change to `worker.ts` itself — the job-processing and BullMQ Worker instantiation stay identical. Per §2.1, the worker's *job-processing logic* is unchanged; this signal handler is part of the *startup harness* explicitly carved out as in-scope for the lifecycle work.

### 3.6 Test discipline

TDD per project default (Vitest, write failing test → see fail → minimum implementation → see pass):

- **Unit tests** at `tests/unit/queue/workerLifecycle.test.ts`. Mock `child_process.fork` and the Redis client. Cover:
  - Redis-up → spawn called with the expected args.
  - Redis-down → `start()` throws `RedisUnavailableError` with the expected message and `cause`.
  - Pending-jobs rejection on simulated `child.on('exit')` non-zero exit — all tracked rejecters called with `WorkerCrashedError(code, signal)`.
  - `releaseJob(jobId)` removes from the registry; subsequent crash does not re-reject.
  - Idempotent `shutdown()` — second call is a no-op, doesn't throw.
  - SIGTERM → SIGKILL escalation timer fires after `killTimeoutMs`; doesn't fire if child exits naturally first.
  - Stdio routing — `child.stdout`/`child.stderr` piped to the configured log path.

- **Integration tests** at `tests/integration/workerLifecycle.test.ts` (`describe.skipIf(!process.env.REDIS_URL)`). Real Redis. Real `fork` of stub workers in `tests/fixtures/workers/` that either exit 0 immediately, exit non-zero immediately, or hang indefinitely. Verify:
  - Orchestrator rejects pending bash promises within 100ms of stub crash (not the BullMQ-default 30s stall).
  - Graceful shutdown completes within the 2s SIGKILL timer when the stub responds to SIGTERM.
  - SIGKILL escalation actually kills the stub when it ignores SIGTERM.

- **Smoke regression.** Re-run v1.4 walk-point 3 (`bash` tool call) without manually starting the worker. Should succeed. Add as `MANUAL_SMOKE.md` walk-point 0 (pre-flight): "Confirm bash tool works with no separate worker terminal running."

### 3.7 Out of scope for Phase 1 (deferred or rejected)

- **Role-to-model dispatcher** — Phase 2 (Anamorph Routing). Bundling here would mix process-management logic with API-routing logic and muddy debugging. Per Gemini's review: keep Phase 1 tightly scoped to lifecycle.
- **Worker auto-restart on crash** — fail-fast at single-dev scale. Future `myceliate doctor` subcommand if real usage data justifies.
- **Per-worker resource limits** — irrelevant at single-dev scale; container-level limits if ever needed.
- **Multiple worker processes / pool** — single worker is sufficient for single-dev concurrency. Pool support is a v2+ concern when/if a deployment story emerges.
- **Worker `/health` endpoint or IPC heartbeat** — premature. The `child.on('exit')` hook covers crashes; periodic health-check IPC adds complexity for a single-dev tool that restarts in seconds.

## 4. Phase 2 — Anamorph Routing (outline)

A Spore-role-to-model dispatcher. Adds `roleToModel(role: SporeRole): 'deepseek-v4-pro' | 'deepseek-v4-flash'` to the orchestrator and routes per-call rather than per-session.

Default mapping:
- **Germination / planning / Spore creation** → V4-Pro (Teleomorph — generative, novel, architectural).
- **Spore subagent execution / mechanical tool loops** → V4-Flash (Anamorph — cloned, mechanical, replicative).
- **Reasoning-trace-required turns** (R2 thinking-mode active) → V4-Pro regardless of role, to preserve cumulative `reasoning_content`.

Strictly additive. No R1–R12 changes — the routing infrastructure (`req.model` plumbing) already exists in v1.4. ~100–200 LOC. Full design lands as a §4 amendment to this doc (per the v1.4 single-spec precedent) before Phase 2 implementation begins.

## 5. Phase 3 — Exoenzyme (outline)

External MCP server schema → lightweight Markdown skill translation. The Fat Platform mediates: it speaks raw MCP to the server but exposes minimal text-based primitives to the Spore, matching the existing skill-loading pattern.

Lifts MCP from the v1 deferred list (CLAUDE.md line 188) but constrains it to the skill-based translation pattern from research brief Vector 6, **not** raw schema injection. Target servers: Playwright (browser automation) + Postgres (DB introspection) as proof-of-concept of the translation pipeline. Full design before Phase 3 begins.

## 6. Phase 4 — Anastomosis (outline)

Agent-controlled `compact_state` tool. The Spore decides when to compact (typically at logical task boundaries) and what to write into the structured summary; the platform owns the safe context-window clear and re-inject of the summary as a fresh anchor document under `.myceliate/memory/anchors.md`.

Reframes R10 layer 4 (ACE virtual projections) and layer 5 (structured 9-section auto-compaction) — both currently in the v1 deferred list — as a single agent-driven primitive, sidestepping the platform-side guess about when to compact. Full design before Phase 4 begins.

## 8. Out of scope and deferred — full list

The strategic research brief identified 10 vectors. v1.5 implements pieces of vectors 1, 2, and 6 (Anastomosis, Tiered Routing, Skill-Based MCP). Explicitly deferred to v1.6+ or later:

- **Vector 4 — Parallel Message Buses.** Per brief: 15× token-burn multipliers, unsettled coordination patterns. Defer.
- **Vector 5 — P2P Multiplayer Collaboration.** Per brief: A2A protocol still maturing. Defer six-plus months.
- **Vector 7 — Local Episodic Memory (FTS5 + Nomic Embed v2).** Medium leverage, large infrastructure lift. Schedule for v1.6 or v1.7.
- **Vector 8 — Cron Spores / OpenClaw Daemon.** "Exceptionally high leverage" per brief but financially gated by Phase 2 (Anamorph Routing) — daemonised polling on V4-Pro would burn budget. Schedule after v1.5 ships, with routing as a hard prerequisite.
- **Vector 9 — MicroVM Sandboxes (Blaxel/E2B).** Optional remote-execution integration. Defer until cron-spore work generates enough untrusted execution traffic to justify.
- **Vector 10 — Multi-modal Vision.** Wait until a UI-automation Spore actually lands (likely a Phase 3 Exoenzyme follow-on).
