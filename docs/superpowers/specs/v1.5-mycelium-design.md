# Myceliate v1.5 — Mycelium Sprint

## 1. Overview & Goal

v1.5 matures the Myceliate network through four sequential phases. The sprint takes its name from the mycelial network itself — an underground web of hyphae that connects, routes, digests, and remembers. Each phase wires another layer of that network into the existing v1.4 architecture, all strictly additive — R1–R12 stay intact.

The four phases:

- **Phase 1 — Hyphal Continuity.** Harden the existing worker lifecycle module (`src/runtime/workerLifecycle.ts`, present since v1.1 Phase 14, called at `src/index.ts:191`) with observability and fast crash detection. The orchestrator *already* auto-spawns the worker as a managed child process via `spawn('pnpm', ['queue:worker'])`; v1.5 routes the worker's stdio to a log file (giving the user a diagnostic source — the v1.4 walk-point 3 "worker not running" report was almost certainly a silent worker crash with no visible log, since the existing module `.resume()`-discards stdio), adds a `redis.ping()` pre-flight (clean diagnostic instead of a confusing 30s queue-add timeout), explicitly injects `REDIS_URL` into the spawned environment, and adds a pending-jobs registry for ~100ms crash detection on bash dispatch (currently `job.waitUntilFinished()` hangs ~30s waiting for BullMQ's stall watcher). Preserves R6 intact; preserves the existing `spawn('pnpm', ['queue:worker'])` invocation pattern.

- **Phase 2 — Anamorph Routing.** Add a Spore-role-to-model dispatcher. Mechanical/cloned execution paths (Anamorph, V4-Flash) and generative/architectural paths (Teleomorph, V4-Pro) get routed deterministically. Cuts bulk operational cost ~92% for terminal-heavy workloads while preserving reasoning quality on planning turns.

- **Phase 3 — Exoenzyme.** Translate external MCP server schemas into lightweight Markdown skills before the agent ingests them — sidecar processes that digest raw schema noise so the Spore stays thin. Target servers: Playwright (browser automation) and Postgres (DB introspection) as proof-of-concept.

- **Phase 4 — Anastomosis.** Agent-controlled `compact_state` tool for semantic handoffs. The Spore writes a structured summary to `.myceliate/memory/anchors.md` at logical task boundaries; the platform safely clears the context window and re-injects the summary as a fresh anchor. Replaces R10's automatic platform-side compaction with agent-aware semantic compaction.

This spec covers Phase 1 in implementation detail. **Phases 2–4 are scoped at the architectural-shape level here and will receive their own design amendments before each phase begins.** The intent is to ship Phase 1 fast, then design and implement the routing/MCP/anastomosis trio sequentially through v1.5.

The strategic research brief (`docs/Autonomous-Agent-CLI-Research-Brief.docx`) informed this sprint shape. The brief recommended Phase 1 = Tiered Routing, but codebase verification showed the routing infrastructure already exists in v1.4 (`model` is plumbed end-to-end through adapters, `subagentLoop`, `replSession`, `index.ts`, `QueryEngine`); only a Spore-role dispatcher is missing. That dispatcher becomes Phase 2. Phase 1 instead addresses a concrete v1.4 lived-experience gap: walk-point 3 silently failing because the BullMQ worker wasn't running in another terminal.

**Out of scope and deferred to v1.6+ or later** (full list in §8): cron spores / OpenClaw daemon (research brief Vector 8); FTS5 episodic memory layer (Vector 7); P2P agent collaboration (Vector 5); parallel message buses (Vector 4); MicroVM sandboxes (Vector 9); raw-vision multi-modal context (Vector 10).

## 2. Architectural Anchors

Three decisions are load-bearing for the Phase 1 implementation; later phases extend these but don't break them.

### 2.1 Extend (don't replace) the existing worker lifecycle module

The orchestrator already owns the worker process lifecycle. `src/runtime/workerLifecycle.ts:18` exposes `startWorker(): WorkerHandle` and is called at `src/index.ts:191`. The existing implementation (since v1.1 Phase 14 at commit `80e5418`) already:

- Spawns `pnpm queue:worker` as a managed child via `node:child_process.spawn` with `detached: false`.
- Pipes `stdout`/`stderr` (avoiding `inherit` per U4) and drains them via `.resume()` so the ~64KB pipe buffer never fills.
- Handles spawn errors (ENOENT/EACCES) via `child.on('error')` with an `stderr` write before Ink mounts.
- Exposes idempotent `shutdown()` with SIGTERM → 2s wait → SIGKILL escalation.

v1.5 **extends** this module — it does not replace it. The genuinely new behaviours are: route stdio to `.myceliate/logs/worker.log` instead of discarding it (the actual fix for v1.4 walk-point 3 — the worker was running but its diagnostic output had nowhere to go); add a `redis.ping()` pre-flight before spawn; inject `REDIS_URL` explicitly into the spawned environment (defense in depth against silent instance-mismatch); track in-flight bash jobs in a pending-jobs `Map<jobId, {reject}>` to enable ~100ms crash detection; add a `shutdownInitiated` flag to distinguish clean shutdown from crash in the exit handler. The `spawn('pnpm', ['queue:worker'])` invocation pattern is preserved — migrating to `child_process.fork('./dist/queue/worker.js')` would force dev/prod path resolution complexity (`import.meta.url` + tsx-loader detection) for no user-visible benefit.

The `src/queue/worker.ts` job-processing logic is unchanged; its existing SIGINT/SIGTERM handler at `worker.ts:27-50` is the contract the lifecycle's escalation relies on (verified in §3.4). R6 (BullMQ for heavy I/O) stays intact.

### 2.2 Pending-jobs registry for crash recovery

The orchestrator's bash tool returns a Promise resolved by BullMQ's `completed` event. If the worker crashes mid-job, BullMQ's default behaviour is to wait for the job lock to expire (~30s) before marking the job stalled. From the user's perspective, the CLI hangs.

To eliminate this hang, the extended `WorkerHandle` exposes `trackJob(jobId, reject)` and `releaseJob(jobId)` methods backed by an internal `Map<jobId, { reject: (err: Error) => void }>` of in-flight bash promise rejecters. The bash tool dispatch in `src/tools/bash.ts:78-87` calls `trackJob` immediately after `queue.add()` and `releaseJob` once `job.waitUntilFinished()` resolves. When `child.on('exit')` fires with a non-zero code (or any exit while jobs are pending), the lifecycle iterates the Map and rejects all pending promises with `WorkerCrashedError`. The orchestrator's ReAct loop surfaces the crash to the user immediately — typically within 100ms — rather than waiting on BullMQ's ~30s stall watcher (or the indefinite `job.waitUntilFinished` hang).

`Map`, not `WeakMap`, because the key is a primitive `jobId: string`. Entries are explicitly released on completion or failure via `releaseJob(jobId)`.

### 2.3 Fail-fast on missing infrastructure

A pre-flight `redis.ping()` runs before the worker is spawned. If Redis is down, the CLI exits with a clean diagnostic — `myceliate: Redis is required for bash execution. Run docker compose up -d redis and try again.` — rather than spawning a worker that crashes immediately on connection failure.

We explicitly do **not** auto-restart the worker on crash. At single-developer scale, auto-restart silently masks environment rot (Redis container drift, OOM patterns, version mismatches). Fail-fast surfaces these conditions where the user can address them. The deferred future is an explicit `myceliate doctor` subcommand that diagnoses worker-process state on demand; auto-restart can be reconsidered if real usage shows the failure modes are predominantly transient.

## 3. Phase 1 — Hyphal Continuity

### 3.1 Extend `src/runtime/workerLifecycle.ts`

Modify the existing module — do not create a new file. The current `WorkerHandle` type and `startWorker()` factory grow new responsibilities; the existing `spawn('pnpm', ['queue:worker'])` pattern, Ink-safe stdio piping, spawn-error surfacing, and shutdown escalation stay intact. Net delta: ~70-100 LOC added to the existing 56-line file.

**Updated TypeScript surface:**

```ts
// existing — extended
export type WorkerHandle = {
  child: ChildProcess;
  /** Register an in-flight bash job; lifecycle calls reject() on worker crash. NEW. */
  trackJob: (jobId: string, reject: (err: Error) => void) => void;
  /** Bash job completed/failed normally — drop from pending registry. NEW. */
  releaseJob: (jobId: string) => void;
  /** Existing — SIGTERM → 2s wait → SIGKILL. Idempotent. Unchanged signature. */
  shutdown: () => Promise<void>;
};

// new errors
export class RedisUnavailableError extends Error {
  constructor(cause?: Error) {
    super(
      'myceliate: Redis is required for bash execution. ' +
      'Run `docker compose up -d redis` and try again.',
      { cause },
    );
    this.name = 'RedisUnavailableError';
  }
}

export class WorkerCrashedError extends Error {
  constructor(
    public readonly exitCode: number | null,
    public readonly signal: NodeJS.Signals | null,
  ) {
    super(`BullMQ worker crashed (code=${exitCode}, signal=${signal})`);
    this.name = 'WorkerCrashedError';
  }
}

// signature change: now async (due to pre-flight ping); requires opts param
export async function startWorker(opts: {
  redisUrl: string;
  logger: Logger;
}): Promise<WorkerHandle>;
```

**Breaking change to one callsite:** the existing `startWorker()` is synchronous and zero-arg. The new signature is async with an opts object. The single caller at `src/index.ts:191` is updated in §3.2.

### 3.2 Updated callsite at `src/index.ts:191`

The existing call is `const worker = startWorker();` (synchronous, zero-arg). It becomes:

```ts
const redisUrl = process.env.REDIS_URL ?? 'redis://localhost:6379';   // matches connection.ts:22 default
const worker = await startWorker({ redisUrl, logger });   // throws RedisUnavailableError → main() catches and exits 1
```

The `redisUrl` resolution duplicates the default in `connection.ts:22`. Optional follow-up: extract `resolveRedisUrl()` helper into `src/queue/connection.ts` and re-use from both spots. Not load-bearing for Phase 1; ride-along candidate if naturally adjacent.

**Behaviours added inside the extended `startWorker()` (in roughly this order in the implementation):**

1. **Redis pre-flight — NEW.** `await getRedis(redisUrl).ping()` with a 2s timeout — hits the existing singleton in `src/queue/connection.ts:22`, no second client created. On failure: throw `RedisUnavailableError`. Caller in `main()` catches and exits 1 with the diagnostic message.

2. **Spawn the worker — invocation pattern unchanged, env extended.** Existing call is `spawn('pnpm', ['queue:worker'], { stdio: ['ignore', 'pipe', 'pipe'], detached: false })`. Becomes `spawn('pnpm', ['queue:worker'], { stdio: ['ignore', 'pipe', 'pipe'], env: { ...process.env, REDIS_URL: redisUrl }, detached: false })`. **The explicit `REDIS_URL` injection is the only change to the spawn call** — it guarantees the worker's `getRedis()` (which reads `process.env.REDIS_URL`) connects to the *exact* instance the orchestrator just verified, not whatever happens to be in the inherited env. Closes the silent-mismatch failure mode where orchestrator pings instance X, worker connects to instance Y, jobs queue to X, never complete, no crash, and the orchestrator hangs forever. **Do NOT migrate to `child_process.fork`** — see §3.7.

3. **Route stdio — replaces existing `.resume()`-discard.** Open `.myceliate/logs/worker.log` as an append stream (`fs.createWriteStream(path, { flags: 'a' })`; create the `.myceliate/logs/` directory if missing). Pipe `child.stdout` and `child.stderr` through it: `child.stdout?.pipe(logStream); child.stderr?.pipe(logStream);`. **This replaces the existing `child.stdout?.resume(); child.stderr?.resume();` lines** at `workerLifecycle.ts:29-30`. The Ink UI never sees worker output (honours U4). **This is the actual user-visible fix for v1.4 walk-point 3** — when bash misbehaves the user now has `tail -f .myceliate/logs/worker.log` to diagnose, where previously diagnostic output was silently dropped.

4. **Existing spawn-error handler stays.** `child.on('error', err => process.stderr.write(...))` at `workerLifecycle.ts:35-37` continues to fire before Ink mounts on ENOENT/EACCES. No change.

5. **Crash handler — NEW.** `child.on('exit', (code, signal) => ...)`. The lifecycle module gains a private `shutdownInitiated: boolean` flag, set to `true` as the **first line** of the existing `shutdown()` body (before the existing `if (shuttingDown) return;` idempotency check). The exit handler distinguishes:
   - `shutdownInitiated === true` → clean shutdown path. Log `{ event: 'worker_shutdown_complete', code, signal }`. No promise rejection.
   - `shutdownInitiated === false` AND (`code !== 0` OR pending jobs exist) → crash path. Iterate the pending-jobs Map and call each `reject(new WorkerCrashedError(code, signal))`. Log `{ event: 'worker_crashed', exitCode: code, signal, pendingJobs: [...keys] }` so the next ReAct iteration surfaces the crash via the orchestrator's existing error-handling path.

6. **Existing `shutdown()` SIGTERM → 2s → SIGKILL escalation stays.** No change to lines 39-54 of the current file beyond the new `shutdownInitiated = true` first line.

7. **Pending-jobs Map — NEW.** Private `pendingJobs = new Map<string, (err: Error) => void>()`. `trackJob(jobId, reject)` sets; `releaseJob(jobId)` deletes. Both methods exposed on the returned `WorkerHandle`.

### 3.3 Bash dispatch integration — `src/tools/bash.ts:78-87`

The current bash dispatch is:

```ts
const toolUseId = randomUUID();
const job = await deps.queue.add(
  'bash',
  { command: input.command, cwd, timeoutMs, toolUseId },
  { jobId: toolUseId },
);
const result = await job.waitUntilFinished(deps.queueEvents);
return formatResult(result);
```

The `job.waitUntilFinished(deps.queueEvents)` call is what hangs ~30s if the worker dies mid-job. To get ~100ms crash detection, race `waitUntilFinished` against a rejection promise registered with the workerLifecycle:

```ts
const toolUseId = randomUUID();
const job = await deps.queue.add(
  'bash',
  { command: input.command, cwd, timeoutMs, toolUseId },
  { jobId: toolUseId },
);
let crashReject!: (err: Error) => void;
const crashPromise = new Promise<never>((_, reject) => { crashReject = reject; });
deps.worker.trackJob(toolUseId, crashReject);
try {
  const result = await Promise.race([
    job.waitUntilFinished(deps.queueEvents),
    crashPromise,
  ]);
  return formatResult(result);
} finally {
  deps.worker.releaseJob(toolUseId);
}
```

`BashToolDeps` (currently `{ hitl, queue, queueEvents, defaultTimeoutMs, ... }` at `bash.ts:16-26`) gains a `worker: WorkerHandle` field. `bootTools()` in `src/runtime/bootTools.ts` already receives `queue` and `queueEvents` from `index.ts` after `startWorker()` returns — thread the awaited `worker` handle through the same way (`bootTools` opts gain `worker`; `bootTools` passes it down to `createBashTool({ ...deps, worker })`).

This is the only orchestrator-side code touched outside `src/runtime/workerLifecycle.ts` and `src/index.ts`. `src/queue/` files are not touched at all.

### 3.4 Worker-side: graceful shutdown signal handling

**Verified — no `worker.ts` changes needed.** `src/queue/worker.ts:27-50` already implements `shutdown()`, registered on both `SIGINT` and `SIGTERM`, with idempotent `shuttingDown` flag protection. It calls `worker.close()` (drains BullMQ in-flight jobs back to the queue), then `events.close()` (closes the QueueEvents subscriber), then `closeRedis()`, then `process.exit(0)`. Each step is wrapped in `try/catch` so a failure in one doesn't leak the rest.

This is the contract the lifecycle module's `SIGTERM → wait killTimeoutMs → SIGKILL` escalation in §3.2 step 4 relies on: the worker responds to SIGTERM by initiating a clean drain; if the drain doesn't complete within the timeout, the lifecycle escalates to SIGKILL. In practice the existing `worker.close()` returns within milliseconds for an idle worker and within job-timeout bounds for a busy one, so the SIGKILL path should rarely trigger in real use.

Per §2.1, the worker's job-processing logic is unchanged. The shutdown handler was already part of v1.4's worker startup harness; v1.5 just inherits it without modification.

### 3.7 Out of scope for Phase 1 (deferred or rejected)

- **Migrating from `spawn('pnpm', ['queue:worker'])` to `child_process.fork('./dist/queue/worker.js')`** — the existing spawn pattern works in both dev and prod via the npm script abstraction (pnpm finds the script; tsx runs it in dev; node runs the built JS in prod). Migrating to `fork` would force `import.meta.url`-relative resolution + tsx-loader detection (`process.execArgv.some(a => a.includes('tsx'))` or a `MYCELIATE_DEV` env flag) for no user-visible benefit. Reconsider in v2 only if the IPC channel becomes desirable for richer worker↔orchestrator messaging beyond what BullMQ already provides.
- **Role-to-model dispatcher** — Phase 2 (Anamorph Routing). Bundling here would mix process-management logic with API-routing logic and muddy debugging. Per Gemini's review: keep Phase 1 tightly scoped to lifecycle.
- **Worker auto-restart on crash** — fail-fast at single-dev scale. Future `myceliate doctor` subcommand if real usage data justifies.
- **Per-worker resource limits** — irrelevant at single-dev scale; container-level limits if ever needed.
- **Multiple worker processes / pool** — single worker is sufficient for single-dev concurrency. Pool support is a v2+ concern when/if a deployment story emerges.
- **Worker `/health` endpoint or IPC heartbeat** — premature. The `child.on('exit')` hook covers crashes; periodic health-check IPC adds complexity for a single-dev tool that restarts in seconds.

## 4. Phase 2 — Anamorph Routing

A Spore-role-to-model dispatcher. Mechanical/cloned execution paths (Anamorph, V4-Flash) and generative/architectural paths (Teleomorph, V4-Pro) are routed deterministically per-call rather than per-session. Cuts bulk operational cost ~92% for terminal-heavy workloads while preserving reasoning quality on planning turns.

The routing infrastructure already exists in v1.4 — `req.model` is plumbed end-to-end through adapters, `subagentLoop`, `replSession`, `index.ts`, and `QueryEngine`. Phase 2 adds **one new module** (`src/runtime/roleToModel.ts`), **one `QueryEngine` method** (`hasRetainedReasoning()`), and **wires two existing wire-touch sites** (`src/orchestrator/reactLoop.ts:48`, `src/spores/subagentLoop.ts:64`). Strictly additive — R1–R12 stay intact.

`src/tools/germinate_spore.ts` is **not** a wire-touch site, despite being in the original Gemini-locked touchpoint list. Verified by `grep -n "client\.stream(" src` — there are exactly two `client.stream(` calls in the codebase, and germinate_spore is not one of them. The germination tool only mutates orchestrator state (`appendSystemPrompt` + `writePin`); whichever role the orchestrator's current REPL turn carries is the role for the API call that *decides* to germinate.

### 4.1 Architectural anchors

Three decisions are load-bearing for Phase 2; they also frame Phase 4 (Anastomosis), since semantic compaction will likely want a `germination` role hook reserved here.

#### 4.1.1 Per-call dispatch, not per-session

Picking a model at session start and threading it through every API call wastes the cost-quality trade-off. A session that opens with planning ("design me a refactor") and continues into mechanical tool loops ("now grep for X, write file Y, run tests") burns Pro pricing on the mechanical tail; a session that opens with execution and grows complex burns Flash quality on the planning shift. V4-Flash and V4-Pro are priced ~10× apart at the bulk-token level, so per-call dispatch is the only mapping that captures the operational savings the routing was designed for.

The decision lives at the point of API call (`client.stream(...)`). Two such call-sites exist: `src/orchestrator/reactLoop.ts:48` (per ReAct iteration in the orchestrator REPL) and `src/spores/subagentLoop.ts:64` (per iteration in subagent subprocess loops). Both consult `roleToModel(role)` immediately before issuing the request.

#### 4.1.2 Centralised env override

`DEEPSEEK_MODEL` env-var override stays. It is the escape hatch for: testing against Ollama / local DeepSeek mocks (research-time workflow), forcing a specific V4 variant during incident response, and bypassing routing temporarily when a new DeepSeek release ships. Per Gemini's Phase 2 review: "an unmissable warn line — `DEEPSEEK_MODEL env var is set. Bypassing Anamorph/Teleomorph role routing and forcing model: <name>`."

Centralisation means: `process.env.DEEPSEEK_MODEL` is read in exactly **one place** — inside `roleToModel()`. The current direct reads at `src/index.ts:52`, `src/spores/subagentLoop.ts:65`, and `src/spores/childProcessSpawn.ts:17` collapse to a single authoritative source. The warn line is fired exactly once per process at boot via an explicit `checkAndWarnEnvOverride()` call inside `main()` — *before* Ink mounts, so writing to stderr is U4-safe. Inside `roleToModel()` itself the env check is silent and stateless — a pure function once warning has been emitted at boot.

`childProcessSpawn.ts:17` keeps `DEEPSEEK_MODEL` in `ALLOWED_ENV_KEYS` so the override propagates from orchestrator to subagent subprocess. The subagent's `roleToModel` call sees the same env value.

#### 4.1.3 Reasoning ratchet via `QueryEngine.hasRetainedReasoning()`

Within an orchestrator REPL session, role decision needs a runtime signal. Per Gemini's review: "is `reasoning_content` in current message history?" — a stateful detection of whether prior turns produced reasoning that R2 will retain in the next API request.

`QueryEngine.hasRetainedReasoning()` returns `true` iff any assistant message in `this.history` has both `tool_calls` and non-empty `reasoning_content`. This matches R2's retention rule exactly: reasoning is preserved only on assistant messages that include tool_calls, so the check distinguishes "reasoning chain in flight" from "purely conversational turns where R2 will drop the reasoning". Implementation reuses `hasToolCalls` and `hasReasoningContent` from `src/adapters/messages.ts:30-38` — both already exist.

Behaviour is a **monotonic ratchet**: once any assistant turn produces retained reasoning, every subsequent iteration in that session uses `repl-with-reasoning` → V4-Pro. The session never downgrades back to Flash mid-session, even if subsequent iterations are mechanical, because losing the cumulative reasoning context mid-chain would defeat R2's purpose. Cost trade-off: a complex session locks into Pro pricing once reasoning starts; users who want a hard reset use a fresh session (no `--resume`).

**Iteration 0 of every session defaults to `repl-with-reasoning` → V4-Pro, regardless of engine state** (planning bias). Per Gemini's round-2 review: the premise of tiered routing is that planning and requirement decomposition land on the frontier model; routing the user's *opening* prompt to Flash would defeat the routing's strategic purpose for any session that opens with planning ("design me a refactor"). After iteration 0 the ratchet engages: if iteration 0 produced retained reasoning (any tool-call assistant turn carrying `reasoning_content`), iteration 1+ stays on Pro per the monotonic rule above; if iteration 0 produced no retained reasoning (e.g. the user said "run tests" and got a single mechanical tool-call turn the model did not reason about, or answered without tool calls so R2 dropped any reasoning), iteration 1+ drops to Flash. Cost trade-off: trivial single-iteration prompts pay Pro pricing for one call. The cost of *missing* a planning context is much higher than the cost of one over-routed mechanical prompt; this default biases toward the safer error mode and removes the need for a speculative length-based heuristic.

### 4.2 New module: `src/runtime/roleToModel.ts`

Sibling to `workerLifecycle.ts` per Gemini's location decision (both are runtime-orchestration concerns; both are imported by the index.ts boot path). New file, not a modification.

**TypeScript surface:**

```ts

/**

 * dispatcher routes mechanical/cloned-execution roles (Anamorph) to V4-Flash
 * and generative/architectural roles (Teleomorph) to V4-Pro.

 * Reserved for explicit future use:
 *   - 'germination' — orchestrator turn invoking germinate_spore (Phase 4
 *     anastomosis hook; today germination is a tool call inside an existing
 *     repl-* role, not its own dispatch site).
 *   - 'orchestrator' — generic orchestrator default for non-REPL flows
 *     (cron spores in v1.6+, programmatic API entry points).
 */
export type SporeRole =
  | 'subagent'
  | 'repl-execution'
  | 'repl-with-reasoning'
  | 'germination'
  | 'orchestrator';

export type AnamorphModel = 'deepseek-v4-flash';
export type TeleomorphModel = 'deepseek-v4-pro';
export type RoutedModel = AnamorphModel | TeleomorphModel;

const ROLE_MAP: Readonly<Record<SporeRole, RoutedModel>> = {
  subagent: 'deepseek-v4-flash', // R8/R9: ephemeral mechanical loop
  'repl-execution': 'deepseek-v4-flash', // tool-loop turn, no retained reasoning
  'repl-with-reasoning': 'deepseek-v4-pro', // R2 cumulative reasoning in flight
  germination: 'deepseek-v4-pro', // strategic spore-discovery / pre-pin
  orchestrator: 'deepseek-v4-pro', // generic orchestrator-side default
};

/**
 * Resolve the model string for a given role.
 *
 * Precedence:
 *   1. `DEEPSEEK_MODEL` env var (if non-empty) — bypasses role routing entirely.
 *      Documented escape hatch for local-mock testing, incident response, and
 *      pre-release variant testing.
 *   2. `ROLE_MAP[role]` — the canonical Anamorph/Teleomorph mapping.
 *
 * Pure function modulo `process.env`. Logger arg is intentionally absent —
 * the env-override warn is fired exactly once at boot via
 * `checkAndWarnEnvOverride()`, not lazily on each call.
 *
 * Returns `string`, not `RoutedModel`, because the env override may be any
 * string (deepseek-reasoner for V3 testing, ollama:* for local mocks).
 */
export function roleToModel(role: SporeRole): string {
  const envOverride = process.env.DEEPSEEK_MODEL;
  if (envOverride !== undefined && envOverride.length > 0) return envOverride;
  return ROLE_MAP[role];
}

/**
 * Boot-time check: emit an unmissable warn if DEEPSEEK_MODEL is set.
 *
 * Called from `main()` in src/index.ts BEFORE Ink mounts, so writing to
 * stderr is U4-safe. Also emits a structured log record via the supplied
 * logger.
 *
 * Idempotent if invoked twice (no-harm second call). Writes are silent
 * when the env var is unset.
 */
export function checkAndWarnEnvOverride(logger: Logger): void {
  const envOverride = process.env.DEEPSEEK_MODEL;
  if (envOverride === undefined || envOverride.length === 0) return;
  const message =
    `DEEPSEEK_MODEL env var is set. Bypassing Anamorph/Teleomorph role routing ` +
    `and forcing model: ${envOverride}`;
  logger.warn({ event: 'deepseek_model_override', model: envOverride });
  process.stderr.write(`[myceliate] ${message}\n`);
}
```

**Why `string` return type rather than `RoutedModel`:** the env override may legitimately be any string the user wants to test against (V3 `deepseek-reasoner`, Ollama, local mock servers). Constraining the return type to the routed sum would force ugly casts at the override path for no safety benefit — the adapter validates model names against the wire endpoint.

**Why pure modulo `process.env`:** `process.env.DEEPSEEK_MODEL` is read on every call (cheap; Node caches). No module-level state. The "warn-once" semantics live in the explicit boot helper, not in a side-effecting `roleToModel` — putting the warn lazily inside the resolver would either need a module-level mutable flag (bad — hidden process-wide state) or fire on every call (bad — log spam).

### 4.3 QueryEngine extension: `hasRetainedReasoning()`

Add one method to `src/orchestrator/QueryEngine.ts` exposing the R2-relevant state without mutating it. Reuses the existing `hasToolCalls` and `hasReasoningContent` predicates from `src/adapters/messages.ts`.

```ts
import { hasReasoningContent, hasToolCalls } from '../adapters/messages.js';

// inside class QueryEngine:

/**
 * True iff any assistant turn in history has both tool_calls and non-empty
 * reasoning_content — i.e., R2 will retain that reasoning_content in the
 * next API request. Used by reactLoop to dispatch repl-with-reasoning.
 *
 * Monotonic during a session: once true, stays true (R2 never drops
 * reasoning from already-tool-call turns). Caller can trust this is a
 * one-way ratchet; no need to debounce.
 */
hasRetainedReasoning(): boolean {
  return this.history.some((m) => hasToolCalls(m) && hasReasoningContent(m));
}
```

The method does not allocate (`Array.some` short-circuits). It runs once per ReAct iteration over a history bounded by the working budget (R12 caps at 200k tokens, far smaller than the data-structure traversal cost).

No breaking change to existing methods or types. Pure addition.

## 5. Phase 3 — Exoenzyme

External MCP server schema → lightweight Markdown skill translation. The Fat Platform mediates: it speaks raw MCP (JSON-RPC over stdio) to the server but exposes minimal text-based primitives to the Spore, matching the existing skill-loading pattern. Lifts MCP from the v1 deferred list (`CLAUDE.md` line 188) but constrains it to the skill-based translation pattern from research brief Vector 6 — **not** raw schema injection. Target servers: Playwright (browser automation) + Postgres (DB introspection) as proof-of-concept.

The architectural commitment is that **one MCP server → one spore**. An MCP-translated spore has the same on-disk shape as a hand-authored spore (manifest + sector `SKILL.md` + per-tool `commands/`), the same germination semantics (`germinate_spore` injects body + pins for project), and the same allowlist contract. The only addition is a new optional `mcp_server` field in the manifest declaring the server's invocation shape; presence of this field flips germination behaviour to "spawn server lazily + register translated tool wrappers in `ToolRegistry`."

Validated by OpenAI's April 2026 Codex `SKILL.md` primitive (Markdown skill, name+description eagerly loaded, body lazy-loaded on use). Codex bundles skills + apps + MCP servers under one "plugin" unit; Myceliate keeps them as one primitive (the spore) with an optional MCP backing — same outcome via a leaner type system.

### 5.1 Architectural anchors

Six decisions are load-bearing for Phase 3.

#### 5.1.1 The spore is the unit of MCP translation

A new top-level "MCP integration" primitive would mean two ways to load capabilities into a session: hand-authored spores via `germinate_spore`, and a parallel MCP-bindings registry. Two registries means two discovery paths, two activation flows, two allowlist mechanisms, two TUIs. The spore primitive already carries everything an MCP integration needs: tier discovery (bundled/user/project), version manifest, sector-level body injected into the orchestrator system prompt, persona roster, per-pack commands, allowlist gating. An MCP-translated spore is the same primitive with one optional manifest field.

The germination contract preserves: `germinate_spore('playwright')` from the orchestrator activates the Playwright spore exactly the same way `germinate_spore('frontend-design')` activates a hand-authored sector — body injected, pin written, allowlist applied. The only difference is what happens *behind* germination for an MCP-backed spore (server spawn, client init, tool registration). The model never sees that difference.

This also means the bundled/user/project tier ordering applies to MCP-translated spores: an organisation could ship a curated MCP-spore at the bundled tier with hand-authored guidance overrides, with the user-tier auto-translated content overridden as needed. Free composability.

#### 5.1.2 Translation is install-time, body persisted to disk, hand-augmentable

Three options were considered:

- **Install-time** — `myceliate mcp install <name>` runs once, fetches the MCP server's `tools/list`, generates SKILL.md + commands/*.md, writes to `~/.myceliate/skills/<name>/`. Subsequent germinations reuse the cached body.
- **Boot-time** — every `myceliate` startup re-queries configured servers and regenerates skills.
- **Germination-time** — first germinate triggers spawn + translation + cache.

Pick **install-time**. (a) Translation is regenerable on demand (`--regenerate` flag, or just `rm -rf ~/.myceliate/skills/<name>` and re-install); the user is in control of when the schema snapshot is taken. (b) The body lives on disk as plain Markdown — the user can hand-augment it (custom tone, domain-specific usage notes, examples) and those edits survive between sessions. (c) No boot-time slowdown — Myceliate boots in milliseconds today; an MCP `initialize` handshake against a Playwright server is on the order of seconds. (d) No cold-start latency on first germinate — the model proposes germination, the platform responds immediately because the body is already on disk; the only lazy step is the server child spawn. Codex's `SKILL.md` design ratifies this — they ship skills as on-disk Markdown for the same reasons.

A `<!-- MYCELIATE: AUTO-GENERATED ABOVE; user notes BELOW are preserved on --regenerate -->` marker splits the SKILL.md body. `--regenerate` overwrites everything above the marker and concatenates user content below. v1.6 may evolve to a richer merge if the marker proves too coarse.

#### 5.1.3 Lazy server spawn on germination, in-process JSON-RPC client

Process model: one MCP server per spore = one OS child process, spawned only when the spore is germinated. No MCP servers run at boot. The JSON-RPC client lives in-process — a single shared connection per germinated spore.

**Why in-process client (not sidecar):** Myceliate is already a single Node process; routing every JSON-RPC call through an additional sidecar IPC just to talk to a server child process adds two hops without isolation benefit. The MCP server *is* the sidecar — it's a separate child process by MCP convention. The MCP SDK's `Client` class with `StdioClientTransport` is purpose-built for this exact shape (`@modelcontextprotocol/sdk/client/stdio`); we use it directly. New dev dependency: `@modelcontextprotocol/sdk` (peer-tested against the 2026 spec revision).

**Lifecycle owner:** `src/runtime/mcpLifecycle.ts`, sibling to `workerLifecycle.ts` (Phase 1). Mirrors that module's discipline: SIGTERM with a grace period (default 2s, `MCP_TEARDOWN_GRACE_MS`), then SIGKILL escalation; explicit child stdio routed to `.myceliate/logs/mcp-<server>.log`; `child.on('exit')` distinguishes clean shutdown from crash; `initialize` handshake with a 5s timeout (`MCP_INITIALIZE_TIMEOUT_MS`).

**Teardown triggers:** (a) `germinate_spore` to a different spore — the previously germinated spore's MCP server, if any, is torn down before the new one spawns; (b) `/spore unpin` slash command; (c) orchestrator process exit (`process.on('exit')` + Ink unmount path call `lifecycle.teardownAll()`). The "no orphaned MCP server children" invariant must hold: a `myceliate` Ctrl+C must not leave a Playwright browser process running.

#### 5.1.4 Sensitive operations route through Cortina HitlGate via `kind: 'mcp'`

The existing `ApprovalRequest` shape is bash-tailored — single `command` field reused as a sentinel for write paths and read paths (Phase 1.5 carry-forward, noted in `feedback_motivated_refactors.md`). MCP-mediated operations (Postgres `execute_query` against a write-capable role, Playwright form fills against auth-gated pages, file-tool MCP servers like `filesystem`) finally motivate the deferred discriminated union refactor.

The new `kind: 'mcp'` variant carries `{ requestId, server, tool, argsSummary, reason }`. The translated SKILL.md's `commands/*.md` entries declare which tools are sensitive via the manifest's `mcp_server.sensitive_tools` allowlist; `SchemaTranslator` threads the flag into each generated wrapper. `HitlGate.checkMcp({...})` intercepts calls to flagged tools and routes through the existing approval UI (the kind:'mcp' branch of `ApprovalPrompt.tsx` renders "MCP server: <server>", "Tool: <tool>", "Arguments: <argsSummary>", plus the reason).

This is feature-motivated cleanup per `feedback_motivated_refactors.md` — the refactor is finally justified by a real consumer (Postgres writes), not speculative extraction. The bash/write/read variants get cleaner internal shapes for free as a side effect.

#### 5.1.5 Pre-filter hook (negative-agent v1.6 forward-compatibility)

`McpClient.callTool()` is the single function every MCP result passes through before becoming a `tool_result` message in the agent's history. v1.6's negative-agent feature (V4-Flash-mediated, intent-aware pruning of large tool outputs — generalisation of R10 `toolOutputPruner.ts` from heuristic to context-aware) plugs in here without changing call sites or wire path.

**No code shipped in Phase 3** — just the architectural commitment to keep `callTool` as the single transformation point for MCP results, and to NOT inline transformations into the wire layer. Concretely, `callTool` returns the raw response; `runReactLoop`'s tool-result wrapping is the existing single chokepoint for any future middleware. The seam already exists; Phase 3 simply does not violate it.

Naturally adjacent because MCP outputs (Playwright accessibility trees of complex pages, Postgres `\dt` on large schemas) are exactly the bloat case the v1.6 feature targets. Recording the seam in this spec keeps v1.6's design free to act on it.

#### 5.1.6 Multi-active MCP spores — single-active body, multi-active tool wrappers

Hand-authored spores are single-active by design: one `BEGIN GERMINATED SPORE` marker in the orchestrator system prompt at any time. Two competing sector contexts confuse the model and dilute focus. That semantic stays intact for MCP-spores' `SKILL.md` body — germinating a new MCP-spore replaces the prior body section via the existing `QueryEngine.replaceGerminatedSection` path.

But MCP-spore germination has a second effect orthogonal to body injection: it registers per-tool wrappers in `ToolRegistry`. Tools, unlike prose guidance, are inherently composable — there is no "competing context" problem when the registry holds both `playwright_navigate` and `postgres_execute_query`. The natural cross-domain workflow (Postgres lookup → Playwright login flow) demands both surfaces be live simultaneously. Per Gemini's review: pure-multi-active (stacking `SKILL.md` bodies in the prompt) violates the spirit of R12 by burning context budget on prose guidance for non-active sectors.

The Phase 3 model is therefore **single-active body, multi-active wrappers**:

- The orchestrator system prompt holds the `SKILL.md` body of the most-recently-germinated spore (existing single-active semantic, preserved).
- Tool wrappers from any MCP-spore germinated this session stay registered until explicit teardown (`/spore unpin`, process exit, or unexpected server crash per §5.5).
- Wrappers from non-active spores are still callable; the model has access to their JSON Schema arg shapes via the API tool definitions, which is sufficient for valid invocation per R3 (DeepSeek's strict-mode JSON Schema is self-documenting for argument structure; prose guidance is an *optimisation*, not a precondition).

Trade-off acknowledged: a model that germinates Playwright, then Postgres, has lost easy reference to the prose-style "when to use Playwright" guidance. The model can re-germinate Playwright to bring its body back; the previously-registered wrappers stay registered (no double-registration — `mcpLifecycle.spawn` is idempotent per §5.5; the `ToolRegistry` pre-existence check in `germinate_spore` skips already-registered wrappers per §5.7).

R12 compatibility: prompt growth stays bounded — at most one body section, regardless of how many MCP-spores have been germinated this session. Token-budget pressure scales with the number of registered tool definitions (their JSON Schema lives in API tool definitions per call), not with germination history.

This anchor supersedes any single-active language elsewhere in §5 for MCP-tool registration (specifically: the original §5.7 step 2's "teardown previous MCP-spore" is removed — see §5.7.2 for the updated germination flow). The single-active semantic remains for hand-authored sector-spore body injection.

```ts
    env: z.record(z.string(), z.string()).default({}),

    call_timeout_ms: z.number().int().positive().optional(), // per-spore override of MCP_CALL_TIMEOUT_MS
  })
  .optional(),
```

When present, the spore is MCP-backed and germination triggers `mcpLifecycle.spawn`. When absent, the spore is hand-authored (existing behaviour — strictly preserved).

`sensitive_tools` is the per-spore allowlist consumed by `HitlGate.checkMcp` per §5.1.4 / §5.8. Names match the MCP tool names exactly (the SDK's `tools/list` response shapes) — **raw names in the manifest, no namespacing**. Namespacing happens at `ToolRegistry` registration time (§5.7) where each tool becomes `<spore-name>_<raw-tool-name>`.

Per-spore scoping: the sensitivity check inside each wrapper closure compares `rawToolName` against THIS spore's `sensitive_tools` list only. Two different MCP servers exposing a tool also called `execute_query` (e.g. Postgres + GraphQL) cannot accidentally cross-trigger each other's sensitivity flag — each lives in its own closure, each compares against its own manifest list. The audit identity passed to `HitlGate.checkMcp` is the `(server, tool)` tuple (per §5.8), never the bare raw name, so the approval UI is unambiguous about which server is being asked to act.

`call_timeout_ms` is an optional per-spore override of the global `MCP_CALL_TIMEOUT_MS` env default (30000ms, per §5.3). Useful when one MCP server's typical operations are systematically slower (Playwright navigation against complex pages) or faster (lightweight introspection) than the global default.

### 5.3 New module: `src/mcp/McpClient.ts`

Thin wrapper over `@modelcontextprotocol/sdk/client` exposing the subset of MCP we use:

```ts
import type { Logger } from '../util/logger.js';

export interface McpToolDescriptor {
  name: string;
  description: string;
  inputSchema: Record<string, unknown>; // raw JSON Schema from server (consumed unchanged by ToolRegistry per §5.7)
}

export interface McpToolResult {
  content: Array<{ type: 'text'; text: string } | { type: 'image'; data: string; mimeType: string }>;
  isError: boolean;
}

export interface McpClient {
  initialize(): Promise<void>;
  listTools(): Promise<McpToolDescriptor[]>;
  callTool(name: string, args: Record<string, unknown>): Promise<McpToolResult>;
  close(): Promise<void>;
  /** Phase 3 hang-protection / crash-detection: true once the underlying child has
   *  exited unexpectedly OR a callTool exceeded callTimeoutMs. Subsequent calls
   *  reject immediately rather than awaiting a dead transport. */
  isFaulted(): boolean;
  /** Phase 3: register a one-shot listener fired when the child exits unexpectedly.
   *  McpLifecycle bridges this to ToolRegistry deregistration + a system-message emit. */
  onUnexpectedExit(handler: (info: { code: number | null; signal: NodeJS.Signals | null }) => void): void;
}

export class McpServerCrashedError extends Error {
  constructor(
    public readonly server: string,
    public readonly exitInfo: { code: number | null; signal: NodeJS.Signals | null },
    override readonly cause?: unknown,
  ) {
    super(`MCP server "${server}" exited unexpectedly (code=${exitInfo.code}, signal=${exitInfo.signal})`);
    this.name = 'McpServerCrashedError';
  }
}

export class McpToolTimeoutError extends Error {
  constructor(
    public readonly server: string,
    public readonly tool: string,
    public readonly timeoutMs: number,
  ) {
    super(`MCP tool "${server}.${tool}" exceeded callTimeoutMs (${timeoutMs}ms) — call abandoned, server may still be working`);
    this.name = 'McpToolTimeoutError';
  }
}

export interface McpClientOpts {
  command: string;
  args: string[];
  env: Record<string, string>;
  initializeTimeoutMs?: number; // default 5000 (env: MCP_INITIALIZE_TIMEOUT_MS)
  callTimeoutMs?: number;       // default 30000 (env: MCP_CALL_TIMEOUT_MS) — Promise.race in callTool guards the ReAct loop from hanging server I/O
  logger: Logger;
}

export function createMcpClient(opts: McpClientOpts): McpClient;
```

The wrapper exists to (a) give us a stable internal interface independent of SDK version churn, (b) provide the `RecordingMcpClient` test-fixture seam (drop-in replacement that captures every call), and (c) own the single-chokepoint property of §5.1.5.

R1 analogue: we do NOT translate JSON-RPC into a "canonical event" intermediate. The SDK's response types pass through as-is. Adding a second MCP server type (HTTP transport, future MCP revisions) means adding another factory; nothing else changes.

**Hang protection (`callTimeoutMs`).** Some MCP server tools execute long-running I/O — Playwright navigations against slow sites, Postgres queries against large tables. Without a timeout, `await client.callTool(...)` deadlocks the orchestrator's ReAct loop. The wrapper enforces `callTimeoutMs` (default 30s) via `Promise.race` against the SDK call. On timeout, the call rejects with `McpToolTimeoutError` carrying the elapsed milliseconds and the (server, tool) identity. The wrapper does NOT cancel the SDK-side promise — the underlying child may still be working and there's no JSON-RPC cancellation primitive in the protocol. A subsequent `client.close()` is the only way to interrupt the child cleanly. The model sees the timeout as a tool_result with `isError: true` and can decide to retry, abort, or change tactic.

**Faulted state on unexpected child exit.** `McpClient` listens to the SDK transport's `'close'` event. On non-clean exit (caller didn't request `client.close()`), the client transitions to a faulted state: `isFaulted()` returns true, all subsequent `callTool` invocations reject immediately with `McpServerCrashedError` rather than hanging on a dead transport. The registered `onUnexpectedExit` handler fires once, with the exit code and signal. `McpLifecycle` (§5.5) listens for this and deregisters the spore's tool wrappers from `ToolRegistry` + emits a system-message-style notification so the model knows the capability is gone.

**Stdio hygiene caveat (inline comment in `McpClient.ts`).** Some misbehaving MCP servers write debug output to stdout instead of stderr, which corrupts the JSON-RPC framing the SDK's `StdioClientTransport` parses. This is a known SDK gotcha. Logs are routed to `.myceliate/logs/mcp-<server>.log` for forensic inspection when an MCP integration breaks unexpectedly. If a particular server proves chronically dirty, a v1.6 ride-along could add a stdout-sniffer that detects non-JSON-prefix lines and routes them to log instead of breaking the stream.

### 5.5 New module: `src/runtime/mcpLifecycle.ts`

Sibling to `workerLifecycle.ts`. Owns the lifecycle of MCP server child processes — one process per germinated MCP-spore, multiple processes alive concurrently per the multi-active model (§5.1.6).

```ts
import type { Logger } from '../util/logger.js';
import type { Spore } from '../spores/Spore.js';
import type { McpClient } from '../mcp/McpClient.js';

export interface McpLifecycleOpts {
  logger: Logger;
  initializeTimeoutMs?: number; // default from MCP_INITIALIZE_TIMEOUT_MS env, fallback 5000
  teardownGraceMs?: number;     // default from MCP_TEARDOWN_GRACE_MS env, fallback 2000
  callTimeoutMs?: number;       // default from MCP_CALL_TIMEOUT_MS env, fallback 30000 (passed through to McpClientOpts)
  /**
   * Phase 3 — invoked when an MCP server child exits unexpectedly (not via
   * teardown()). Bridge for ToolRegistry deregistration + a system-message
   * notification to the model. Without this hook, dead transports stay
   * registered as zombie wrappers and the next callTool invocation rejects
   * deep inside McpClient with no surface visibility.
   */
  onUnexpectedExit?: (sporeName: string, exitInfo: { code: number | null; signal: NodeJS.Signals | null }) => void;
}

export class McpLifecycle {
  constructor(opts: McpLifecycleOpts);
  async spawn(spore: Spore): Promise<McpClient>;
  async teardown(sporeName: string): Promise<void>;
  async teardownAll(): Promise<void>;
  getActive(sporeName: string): McpClient | undefined;
  /**
   * Phase 3 — list all currently-spawned MCP-spore names. Phase 4 (Anastomosis)
   * will use this from compact_state to know which capabilities to re-establish
   * after a context clear (re-germinate the same spores → wrappers re-register
   * via spawn()'s idempotency check).
   */
  listActive(): string[];
}
```

`spawn`:
1. Validate `spore.manifest.mcp_server` is present (throws if absent — caller bug).
2. **Idempotency**: if `getActive(spore.name)` already returns a client (this spore was germinated earlier in the same session and not torn down), return that existing client. Multi-active per §5.1.6 — re-germinating the same spore is a no-op for the lifecycle layer.
3. `createMcpClient({ command, args, env, callTimeoutMs: spore.manifest.mcp_server.call_timeout_ms ?? opts.callTimeoutMs, ... })`. Per-spore override of `callTimeoutMs` (per §5.2) takes precedence over the lifecycle default.
4. `await client.initialize()` with timeout; on timeout, `close()` and reject.
5. Register child-exit listener via `client.onUnexpectedExit(...)`. The listener (a) invokes `opts.onUnexpectedExit(spore.name, exitInfo)` if set, (b) removes the entry from the active map.
6. Register entry in `Map<sporeName, { client; childPid; spawnedAt: number }>`.
7. Return client.

`teardown`:
1. `client.close()` — closes stdio transport, sends shutdown notification (clean-exit signal so the unexpected-exit handler does NOT fire for this teardown).
2. SIGTERM the child.
3. `setTimeout(grace)` → SIGKILL if not yet exited.
4. Deregister from map.

`teardownAll`: iterate active map, await each `teardown` in parallel. Settle-all semantics — one spore's teardown error does not abort the rest.

**Wired into (full wire-touch list — every site §5.5 depends on; subagent review caught these gaps in the original draft):**

- **`src/runtime/bootTools.ts`** — constructs `McpLifecycle`, threads it into `BootToolsOpts.mcpLifecycle`, and wires `onUnexpectedExit` to a closure that calls `tools.deregisterByPrefix('<spore-name>_')` (per §5.7.1's contract) plus emits a system-message StreamEvent so the model knows the capability dropped. The same closure is reused as the deregistration path for explicit teardown (returned from `bootTools()` as `teardownMcpSpore`, per §5.7.3).
- **`src/tools/germinate_spore.ts`** (§5.7.2): `await lifecycle.spawn(spore)` BEFORE injecting the SKILL.md body so the model never sees a body referring to tools that aren't yet registered. With multi-active (§5.1.6), no teardown of prior MCP-spores happens here.
- **`src/runtime/replSession.ts:151-155`** (`/spore unpin` handler) and **`src/index.ts:290-311`** (`onActiveSporeChange` callback): when the user unpins, invoke `bootTools`'s `teardownMcpSpore(sporeName)` closure. With the multi-active model (§5.1.6), `/spore unpin` is the **primary** teardown trigger — germination no longer tears down prior MCP-spores.
- **`src/index.ts:439-455`** `finally` block: insert `await lifecycle.teardownAll()` BEFORE `worker.shutdown()`. Without this, orphaned MCP server children outlive the orchestrator on `/quit` or any fatal error path. Walk-point 10 step 7 verifies the invariant.

R6 note: `spawn()` and `initialize()` are NOT routed through BullMQ — they're orchestrator-side lifecycle operations, not bash dispatch. Subsequent `client.callTool` calls are not bash either — they're stdio JSON-RPC, single message in / single message out, no shell expansion. The `callTimeoutMs` guard (per §5.3) protects against long-running server I/O without needing the BullMQ crash-detection machinery R6 provides for shell commands. R6 stays intact.

Today's argv parsing (`src/index.ts:54-56`) is flag-only (`parseResumeFlag`, `parseNoSporeFlag`).   | { kind: 'mcp-install'; name: string; command: string; args: string[]; env: Record<string,string>; regenerate: boolean };


The two existing call-sites at `src/index.ts:54-56` collapse to a single `const sub = parseSubcommand(process.argv.slice(2));` followed by a discriminated dispatch (`if (sub.kind === 'mcp-install') return runMcpInstall(sub); else proceed with interactive boot using sub.resumeId / sub.noSpore`). Without subsumption, two separate parsers run in sequence and `myceliate mcp install foo --resume bar` gets misparsed.

The `mcp install` shape:

```
myceliate mcp install <name> --command <command> [--arg <arg>]... [--env KEY=VAL]... [--regenerate]
```

Example:

```
myceliate mcp install playwright --command npx --arg @playwright/mcp@latest
myceliate mcp install postgres --command npx --arg @modelcontextprotocol/server-postgres --arg postgresql://localhost/mydb
```

**Atomic install flow** (in a new `src/cli/mcpInstall.ts`). Per Gemini's review: file writes must be atomic — partial failure must not leave a half-written skill directory.

1. Resolve target dir: `~/.myceliate/skills/<name>/`. If exists and `--regenerate` not passed, exit with error.
2. **Stage to a temp dir** (`~/.myceliate/skills/.staging/<name>-<timestamp>/`). All file writes go to the staging dir, never to the target dir directly. Wrapped in try/catch — any error in steps 3–8 triggers `await rm(stagingDir, { recursive: true, force: true })` and the user sees a clean failure with no half-written skill directory left behind.
3. `createMcpClient({ command, args, env, ... })` — temporary client, lifecycle owned locally.
4. `await client.initialize()`.
5. `const tools = await client.listTools()`.
6. `await client.close()`.
7. Construct manifest: `{ name, description: <derived from server initialize info>, version: '0.1.0', accent_color: <derived>, keywords: [], agents: [], allowed_tools: tools.map(t => '<name>_' + t.name), mcp_server: { command, args, env, sensitive_tools: [] } }`. Note `allowed_tools` carries the **namespaced** form (matches what gets registered in `ToolRegistry` per §5.7).
8. Run `translateMcpSchema(tools, manifest)`. Write `manifest.yaml`, `SKILL.md` (frontmatter + skillBody + auto-gen marker + below-marker user-content if `--regenerate`), `commands/*.md` — all into the staging dir.
9. **Atomic move**: `rename(stagingDir, targetDir)` — POSIX `rename(2)` is atomic on the same filesystem. The user's `~/.myceliate/skills/` and `~/.myceliate/skills/.staging/` share a parent (so same filesystem in any sane setup), and the rename is observed as instantaneous.
10. Print to stdout: paths written, capability count, "now run `myceliate` and germinate <name>".

`--regenerate` reads any existing target SKILL.md, preserves the below-marker portion, and writes the merged result into staging before the atomic move. User-edited `commands/*.md` files are NOT preserved (regenerate writes the canonical versions) — users wanting custom command bodies should edit SKILL.md instead. Documented behaviour.

No bundled curated registry of MCP servers in Phase 3. User supplies the command directly. v1.6+ may layer a curated alias (`myceliate mcp install playwright` → known-good command lookup) once the user-supplied-command pattern proves out.

### 5.7 Wire sites: `ToolRegistry` extensions + `germinate_spore.ts` + `bootTools.ts` threading

#### 5.7.1 Supporting change: `ToolRegistry.deregister` and dual-schema `Tool<Input>`

Two extensions to `src/tools/registry.ts`, both small but load-bearing:

**(a) Deregister methods.** The current registry has `register(tool)` but no inverse. Phase 3 needs deregistration on (i) explicit `/spore unpin <name>` (per §5.5), (ii) MCP server crash via `onUnexpectedExit` (per §5.5). Add:

```ts
/**
 * Phase 3: remove a tool from the registry. No-op if name is unknown.
 * The active allowlist is NOT mutated — if a deregistered name was in the
 * allowlist, getActiveTools() will silently exclude it next call (the filter
 * intersection naturally handles this). Re-registering the tool later
 * (re-germinating the spore) restores it without any allowlist update.
 */
deregister(name: string): void;

/**
 * Phase 3: bulk variant for spore teardown. Removes all tools whose name
 * starts with `prefix` (e.g. 'playwright_'). Returns the count removed for
 * the caller to use in user-facing messages.
 */
deregisterByPrefix(prefix: string): number;
```

**(b) Dual-schema tool support.** Per Gemini's review (#2), MCP tool wrappers should NOT round-trip through a JSON Schema → Zod converter. The current `Tool<Input>` requires `inputSchema: z.ZodType<Input>` because `ToolRegistry.definitions()` calls `zodToStrictJsonSchema` to produce the API tool definition. For MCP tools, the JSON Schema comes directly from the server's `tools/list` response — already in the exact shape DeepSeek wants.

Extend the `Tool<Input>` shape with a discriminated `inputSchema` field:

```ts
export type Tool<Input> = {
  name: string;
  description: string;
  capability: Capability;
  inputSchema:
    | { kind: 'zod'; zod: z.ZodType<Input> }
    | { kind: 'json-schema'; jsonSchema: Record<string, unknown> };
  run: (input: Input, ctx: ToolRunContext) => Promise<string>;
};
```

`ToolRegistry.definitions()` switches on `kind`:
- `'zod'` → call `zodToStrictJsonSchema(t.inputSchema.zod)` (existing path, native tools).
- `'json-schema'` → return `t.inputSchema.jsonSchema` directly (MCP tools).

`ToolRegistry.invoke()` validation:
- `'zod'` → `t.inputSchema.zod.parse(rawInput)` (existing path).
- `'json-schema'` → trust the model's structured output and pass `rawInput as Input` directly. The MCP server validates internally and returns `isError: true` on bad shapes. Trade-off accepted: a fast/cheap path that delegates validation to the server, vs. a heavyweight in-process JSON Schema validator that we'd have to maintain. v1.6 may add a thin runtime validator if real-world bad-args rates prove problematic.

All existing native tools wrap their schema with `{ kind: 'zod', zod: ... }` — a one-line refactor at each `register(...)` call site (in `bootTools.ts` for the orchestrator, plus the bash-stub variant). No behaviour change for native tools.

**No `jsonSchemaToZod.ts` module is created.** The earlier draft proposed one as a converter; the dual-schema approach makes it unnecessary. Test discipline (§5.9) drops the corresponding `jsonSchemaToZod.test.ts` accordingly.

### 5.12 Out of scope for Phase 3 (deferred or rejected)

- **MCP resources and prompts.** The MCP protocol has three primitives — tools (covered in Phase 3), resources (file-like read endpoints), prompts (parameterised prompt templates). Resources and prompts are deferred. - **In-IDE plugin host.** Codex's plugins run inside their desktop super-app. Myceliate is CLI-only; plugins are MCP-translated spores, no IDE host. Multi-surface positioning is a v2+ strategic question (per the Codex research synthesis); Phase 3 stays CLI-scoped.

### 6.5.2 Caveman mode

New module `src/runtime/cavemanMode.ts` exports:

```ts
export const CAVEMAN_SYSTEM_PREFIX: string;
export type CavemanState = { active: boolean };
export function defaultCavemanState(env: NodeJS.ProcessEnv): CavemanState;
export function isCavemanEnabledByEnv(env: NodeJS.ProcessEnv): boolean;
export function applyCavemanPrefix(messages: Message[], state: CavemanState): Message[];
```

`CAVEMAN_SYSTEM_PREFIX` is a single ~250-token system instruction framing the model as a terse-character: drop articles, drop pleasantries, drop problem-restatement, drop unsolicited explanations, keep code blocks intact, keep markdown for tables and lists. Wording cribbed from the upstream `JuliusBrussee/caveman` conventions, adapted for Myceliate.

`applyCavemanPrefix` is a pure function: if `state.active`, it prepends a new `{ role: 'system', content: CAVEMAN_SYSTEM_PREFIX }` message to the array; otherwise returns the array unchanged. Called at `QueryEngine.prepareRequest` time after the existing system-prompt assembly but before the messages are handed to the adapter. Symmetric application to subagent dispatches via `runSubagentLoop` — both benefit.

State lives on `ReplSessionOpts` as a new mutable `cavemanState: CavemanState` field, mutated only by the slash-command dispatcher. Initial value at boot: `defaultCavemanState(process.env)` which inspects `MYCELIATE_CAVEMAN`. Truthy values (`1`, `true`, `on`, `yes`) enable; everything else disables.

Slash command `/caveman` (no args = toggle), `/caveman on` (force on), `/caveman off` (force off). Emits a `caveman_toggled` log event `{ event: 'caveman_toggled', active: boolean, source: 'slash' | 'env-init' }` on every state change. The dispatcher rendering shows `caveman ON` or `caveman OFF` to the user via the existing slash-output channel.

### 6.5.3 Cost calculator + DeepSeek pricing

New module `src/runtime/costCalculator.ts`:

```ts
export const PRICING = {
  'deepseek-v4-pro':   { inputPerM: 0.27,  outputPerM: 1.10, cacheHitPerM: 0.07 },
  'deepseek-v4-flash': { inputPerM: 0.014, outputPerM: 0.28, cacheHitPerM: 0.0035 },
  'deepseek-reasoner': { inputPerM: 0.55,  outputPerM: 2.19, cacheHitPerM: 0.14 },
} as const;

export type UsageStats = {
  inputTokens: number;
  outputTokens: number;
  cachedInputTokens?: number;
};

export type CostBreakdown = {
  model: string;
  inputCost: number;
  outputCost: number;
  cacheHitCost: number;
  totalCost: number;
};

export function calculateCost(model: string, usage: UsageStats): CostBreakdown;
export function formatCostUSD(usd: number, places?: number): string;
```

Pricing constants are version-controlled and hand-updated when DeepSeek revises rates. Out-of-list models (env overrides, future adapters) fall back to a zero-cost breakdown with a one-time `cost_unknown_model` warn log line per session.

`calculateCost` is pure. Token counts come straight from the stream's terminal delta. The cache-hit pathway subtracts cached tokens from total input to avoid double-counting: effective input billed at `inputPerM` is `(inputTokens - cachedInputTokens) * inputPerM / 1_000_000`; cache hits billed separately at `cachedInputTokens * cacheHitPerM / 1_000_000`.

`formatCostUSD` returns `$0.0023` (4 decimal places default; 2 dp for session totals where the long tail rounds to noise).

### 6.5.4 Streaming usage extraction

DeepSeek's SSE wire emits a `usage` payload on the final delta (V3 + V4 both, distinct field shapes). The existing `done` event in `src/adapters/streamEvent.ts` has no usage field. Task T34 extends it:

```ts
// Existing:
// { type: 'done' }
// New:
{
  type: 'done';
  usage?: {
    inputTokens: number;
    outputTokens: number;
    cachedInputTokens?: number;
  };
}
```

V3 adapter (`src/adapters/v3/`) and V4 adapter (`src/adapters/v4/`) both populate the field from their respective wire shapes. Existing event consumers that don't look at `usage` continue to work unchanged.

The reactLoop's stream-event consumer captures `done.usage`, calls `calculateCost(model, usage)`, and emits a structured log entry `{ event: 'cost_estimated', model, role, iter, ...usageStats, ...costBreakdown }` per orchestrator iteration. The subagentLoop emits the same shape with `role: 'subagent'` and `step` instead of `iter`. Logger plumbing reuses the seam established by Phase 2's instrumentation closure (`3916c86`).

The same data feeds the Ink layer: the App component subscribes to `cost_estimated` events from a pass-through callback (`onCostEstimate?: (entry) => void` on ReactLoopOptions, mirroring the existing `onState` pattern). The UI accumulates per-turn cost and session total in React state and re-renders the footer.

### 6.5.6 Subagent progress

`src/adapters/streamEvent.ts` gains a new variant:

```ts
{
  type: 'subagent_step';
  step: number;
  durationMs: number;
  model: string;
}
```

`runSubagentLoop` emits the event at the start of each step (after the model is resolved, before the stream call) and on step completion. Subagent process flows through the existing JSON stdout protocol: the subagentRunner's response shape extends to include a `progress: Array<{ step, durationMs, model }>` field alongside the existing `summary`. Minimal version: progress events fire at step boundaries only, not character-by-character. The footer shows `subagent: step N (Ts)` while a subagent is active, blanks when it isn't.
